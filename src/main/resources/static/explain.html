The issue

An overwhelmed API doesn't get slow — it tips over

Threads pile up on a stuck dependency, memory balloons with buffered work, and one noisy client degrades service for everyone. Four failure modes recur. Each POC is the in-process control that prevents one of them.

Failure modeWhat it looks likePattern
Overload A burst or abusive client saturates CPU, pools, downstreams. Rate limiting
Redundant work The same intent arrives many times (keystrokes, spam-clicks, dup webhooks). Debounce / throttle
Unbounded buffering Accept more than we can process → latency grows without bound → OOM. Worker pool + queue
Cascading failure A dead downstream stalls threads until the whole service is down. Breaker + retry
POC 1 · Overload

Rate limiting

Hard problem: decide allow/deny per client, at high concurrency, in microseconds — without a global lock that becomes the bottleneck. And "too many" has two honest definitions: burst-tolerant vs. a strict cap.

Key tech by responsibility

JobMechanism
Per-key isolationConcurrentHashMap + computeIfAbsent
Lock per key, not globalsynchronized(bucket)
Refill / expiry clockSystem.nanoTime() (monotonic)
No timer threadslazy refill from elapsed time

Protects: downstream capacity & CPU from bursts; fairness — one caller can't starve the rest (state is per-key).

Tech debt: node-local (limit × replicas) · unbounded key map (no eviction → leak) · window-log stores every timestamp · remoteAddr is the LB behind a proxy · no Retry-After.
POC 2 · Redundant work

Debounce & Throttle

Hard problem: collapse a flood of rapid events into the right amount of work without losing the caller's latest intent — fire once after quiet (debounce, trailing edge) vs. at most once per interval (throttle, leading edge).

Key tech by responsibility

JobMechanism
Deferred, cancellable fireScheduledExecutorService (1 slot)
Atomic cancel+record+reschedulesynchronized + cancel(false)
Exactly-once gate, lock-freeAtomicLong.compareAndSet

Protects: an expensive action (search query, autosave, webhook) from running per-keystroke / per-click; preserves last-write-wins.

Tech debt: not keyed (all users collapse into one) · debounce belongs on the client (server-side still pays a round-trip/keystroke) · throttle has no trailing fire · scheduler never shut down · node-local.
POC 3 · Unbounded buffering

Worker Pool / Job Queue — backpressure

Hard problem: absorb bursty submissions with bounded resources, and when the burst exceeds capacity, shed load fast instead of buffering forever. Unbounded queueing turns a fast honest failure into a slow catastrophic one.

Key tech by responsibility

JobMechanism
Bounded buffer = backpressure lineArrayBlockingQueue(cap)
Reject-now, don't block calleroffer() (non-blocking)
Cheap workers that block on I/Ovirtual threads
Cap live concurrencyfixed WORKERS loops
Graceful shutdownvolatile running + poll(timeout)

Protects: heap (bounded), the downstream from unbounded concurrency, and the producer's ability to react (fast 503, not a hang).

Tech debt: not durable (crash drops jobs) · jobs map grows forever · no retry/DLQ/priority · 503 without Retry-After · queue is per-replica.
POC 4 · Cascading failure

Circuit Breaker & Retry with Backoff + Jitter

Hard problem: when a downstream is failing, stop making it worse and stop letting it take you down. Breaker: recognise sustained failure and stop calling. Retry: recover from transient blips without a synchronized stampede. They pull opposite ways, so compose with care.

Key tech by responsibility

JobMechanism
Lock-free state flipsAtomicReference<State> + CAS
Exactly one probe (HALF_OPEN)compareAndSet(OPEN, HALF_OPEN)
Lazy OPEN→HALF_OPENcooldown check on read
Bounded backoffmin(max, base·2ⁿ)
Decorrelate clientsThreadLocalRandom jitter

Protects: the downstream from a thundering herd during an outage; our request threads from piling up on a dead dependency; end-to-end latency (fail fast beats hanging).

Tech debt: consecutive-count (not rolling ratio) — a 50%-failing endpoint may never trip · retry blocks the thread, no retry budget → amplification · retries any exception (even fatal 4xx) · per-instance view.
The catch

All of this is node-local — and that breaks at scale

Every pattern keeps state in a field of a singleton bean. Correct in one JVM; wrong-by-default across N replicas (k8s pods / VMs) behind a load balancer. Each replica sees only ~1/N of the traffic, so any algorithm that assumes it sees every event drifts.

PatternHow it drifts across podsFix — where state should live
Rate limit Limit multiplies by N; HPA 3→10 pods silently triples it. Gateway/edge decides once, or Redis atomic Lua counter.
Debounce / throttle Fires once per pod; throttle admits up to N per interval. Move debounce to the client; throttle via Redis SET NX PX.
Worker pool Concurrency = N × workers; backpressure dodgeable by retrying to another pod; not durable. Externalize to a durable broker (Kafka/SQS/SKIP LOCKED) + KEDA autoscale on lag.
Breaker / retry Each pod trips independently (slow herd-thinning); no shared retry budget → N× amplification. Service mesh outlier detection + retry budget; keep jitter.
Why k8s makes it subtle: HPA changes N at runtime (any "global/N" scheme silently rescales) · pods are ephemeral (heap state vanishes on reschedule) · session affinity is a trap, not a guarantee · clock skew appears once timestamps are shared · rolling deploys run two state-schema versions at once.
Rule of thumb: if a pattern protects this pod's own resource (its threads, its heap), node-local is fine. If its correctness depends on a global count or a single global decision, that state must live somewhere all pods share — a store, a broker, the mesh, or the edge — never a pod's heap.
